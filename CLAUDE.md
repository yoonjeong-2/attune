# CLAUDE.md — 프로젝트 기획 문서

> 이 문서는 프로젝트의 모든 핵심 결정을 담고 있습니다.
> 코드를 작성하기 전에 반드시 이 문서를 읽고, 여기 적힌 방향과 범위를 따라주세요.
> 이 문서에 없는 새 기능을 임의로 추가하지 말고, 애매하면 먼저 질문해주세요.

---

## 1. 서비스 한 줄 정의

내 이력서·포트폴리오와 채용공고를 나란히 놓으면, **'이 공고의 눈'으로**
내 서류의 강점·약점을 짚고 무엇을 고칠지 알려주는 웹 서비스.

- 핵심은 "이력서를 잘 고쳐주는 범용 첨삭"이 아니라, **"이 공고 하나와 내 서류가 맞는지"를 판단하는 매칭 분석**이다.
- 반응형 웹 (데스크톱 + 모바일 모두 대응).

---

## 2. 타깃 사용자

- **주 타깃:** 국내 IT 직군(개발·기획) 이직·취업 준비자. 특히 주니어.
- **대표 페르소나:** 김도현, 28세, 백엔드 개발자 2년차, 첫 이직 준비.
  자기 강·약점이 스스로 안 보이는 "자기 객관화" 문제를 겪음.
- **보조 페르소나:** 박서연, 31세, 서비스 기획자(PM) 3년차.
  강점은 알지만 그것을 이 공고에 연결하지 못하는 문제를 겪음.
- 비전은 IT 텍스트 직군 전반으로 열어두되, MVP는 개발·기획 중심.

---

## 3. 해결하는 핵심 문제 (세 가지)

1. **기준의 부재** — 공고의 진짜 핵심 요구가 뭔지 스스로 못 읽음.
2. **자기 객관화 실패** — 내 서류의 강점·약점이 스스로는 안 보임.
3. **피드백 단절** — 탈락 이유를 모른 채 같은 서류로 반복 지원.

→ 이 셋이 합쳐져 "확신 없이 지원 → 붙을 수 있던 기회를 반복해서 잃는" 악순환.

---

## 4. 핵심 기능 (MVP — 딱 이 세 가지)

세 기능은 별개가 아니라 **하나의 분석 흐름**으로 이어진다.

1. **공고 핵심 요구 추출** — 공고에서 '필수'와 '우대'를 분류해 핵심 요구를 짚어줌.
2. **강점·약점 매칭 분석 (서비스의 심장)** — 내 서류와 공고를 대조해,
   강점(어떤 요구와 왜 맞는지)과 약점/공백(왜 부족한지)을 **근거와 함께** 제시.
3. **구체적 개선 제안** — 약점으로 짚인 항목 바로 아래에 "어떻게 고칠지" 제안을 붙임.

---

## 5. 매우 중요한 설계 철학: "정직한 진단"

- **무조건 약점을 만들어내지 말 것.** 서류가 공고에 잘 맞으면 억지 트집을 잡지 않는다.
- 약점은 "반드시 N개"가 아니라 **"있으면 표시, 없으면 없다고 정직하게 말한다"**.
- 프롬프트 설계 시: "약점 3개를 반드시 찾아라"가 아니라
  "공고 대비 실제로 부족한 부분이 있으면 짚고, 없으면 없다고 말하라"로 지시.
- 지적은 3단계로 구분: 치명적 미스매치(필수 요건 없음) / 보완하면 좋은 점(우대 관련) /
  단순 강화 제안(이미 괜찮지만 더 돋보일 수 있음).

---

## 6. 입력 설계 (실체 기준 — 매우 중요)

입력을 "형식(파일/URL/텍스트)"이 아니라 **"추출 가능한 텍스트냐 vs 이미지냐"** 기준으로 처리한다.

### 6-1. 내 서류(이력서·포트폴리오) 입력
- **텍스트 직접 붙여넣기** → 지원 (기본)
- **HTML 링크(노션·웹 이력서·깃허브 등)** → 지원 (텍스트 추출)
- **이미지 기반(이미지 PDF, 구글드라이브 PDF 링크 등)** → MVP 미지원.
  fallback 안내: "이미지 형식은 아직 지원되지 않아요. 노션/웹 링크나 텍스트를 넣어주세요."

### 6-2. 채용공고 입력
- **공고 URL 붙여넣기** → 1순위 (되는 사이트에서 텍스트 추출)
  - 단, 봇 차단·로그인·CORS로 실패할 수 있음. 실패 시 아래 fallback으로 자동 전환.
- **텍스트 직접 붙여넣기** → 안전망
- **이미지 공고** → MVP 미지원, 안내로 처리 (로드맵: OCR)
- 입력창 안내 문구 예: "채용공고 URL을 붙여넣거나, 공고 내용을 직접 붙여넣어 주세요."

### 6-3. 확장성 원칙 (중요)
- **입력을 텍스트로 변환하는 부분**과 **그 텍스트를 분석하는 부분**을 반드시 분리할 것.
- 이렇게 해야 나중에 이미지 OCR/멀티모달을 "앞단에 변환기 추가"만으로 확장 가능.

## 7. 화면 구성 (4개 + 결과 화면 분기)

### 화면 1 · 입력
- 내 서류 입력 + 채용공고 입력을 나란히. 각각 텍스트/링크 지원.
- 이미지 입력 시 fallback 안내.
- "분석하기" 버튼.

### 화면 2 · 분석 중 (로딩)
- 단순 스피너가 아니라 진행 단계 표시: "공고 요구 추출 중 → 서류 대조 중 → 강·약점 정리 중".

### 화면 3 · 결과 (핵심 화면)
- 맨 위에 **전체 적합도 + 한 줄 총평**을 먼저 보여줌.
- 적합도에 따라 아래 구성이 달라짐:
  - **적합도 높음** → 강점 중심으로 풍성하게. 약점 없으면 "치명적 약점 없음" 정직 표시.
  - **적합도 보통·낮음** → 보완점 중심. 각 보완점 **바로 아래 개선 제안**을 붙임(합침).
- 공통 구성: 공고 핵심 요구(필수·우대), 강점(근거 포함).
- 강점/약점 항목은 "공고의 어떤 요구와 연결되는지" 근거를 함께 표시.

### 재진입 (선택)
- 다른 공고로 다시 분석하는 경로.

### 제외 (MVP에 넣지 말 것)
- 로그인/회원가입, 저장/히스토리, 여러 공고 비교, PDF 페이지별 주석, 이미지 처리.

## 8. 기술 스택

- **프론트엔드:** React (반응형).
- **스타일/디자인 시스템:** Tailwind CSS + shadcn/ui.
- **AI 분석:** Claude API 사용. (API 키는 코드에 하드코딩하지 말 것 — 환경변수/서버 경유)
- **백엔드:** 최소한으로. AI 호출과 URL 페이지 불러오기(필요 시)를 위한 얇은 서버 계층만.
  - URL 페이지 불러오기는 CORS 때문에 프론트 단독으로 안 되므로, 서버(프록시) 경유가 필요.
- 구조 원칙: "입력 변환" 계층과 "분석" 계층을 분리 (6-3 참고).

---

## 9. 디자인 방향

- **미감:** 밝고 미니멀. 국내 서비스(토스·당근)의 톤을 참고 — 넉넉한 여백, 큼직하고 명확한 타이포, 절제된 색.
- **라이트 모드가 기본(디폴트).** 다크 모드는 토글로 지원.
- **색:** 흑백/무채색 기반 + 포인트 컬러 최소.
  의미색은 별도로: 강점 = 초록 계열, 약점/보완 = 주황 계열, 적합도 = 상태에 따라.
- 둥근 모서리, 폰트 한 종류로 통일.
- 정보가 많은 결과 화면일수록 **정보 위계와 여백**으로 복잡해 보이지 않게 (미니멀 유지).

## 10. AI 프롬프트 설계 지침 (분석 기능)

- 입력: (1) 내 서류 텍스트, (2) 채용공고 텍스트.
- 출력: 구조화된 형식(JSON 권장)으로 아래를 반환.
  - `overall_fit`: 적합도 (예: 높음/보통/낮음 + 한 줄 총평)
  - `job_requirements`: 공고 핵심 요구 (필수/우대 분류)
  - `strengths`: 강점 배열 (각 항목: 내용 + 연결되는 공고 요구 + 근거)
  - `weaknesses`: 약점/보완점 배열 (각 항목: 내용 + 왜 약점인지 + 개선 제안).
    **없으면 빈 배열.** 억지로 채우지 말 것.
- "정직한 진단"(5번) 원칙을 프롬프트에 명시할 것.
- IT 직군(개발·기획) 맥락을 이해하도록 프롬프트에 직군 컨텍스트를 제공.

---

## 11. 개발 순서 (한 번에 다 만들지 말 것)

1. 프로젝트 뼈대 + 4개 화면 라우팅 (빈 화면).
2. 입력 화면(화면 1) 완성 — 텍스트/링크 입력, fallback 안내.
3. AI 분석 연결 (핵심) — 입력→변환→Claude API→구조화된 결과.
4. 결과 화면(화면 3) — 적합도 분기, 강점/약점/개선, 정보 위계.
5. 로딩 화면(화면 2), 다크모드 토글, 반응형 다듬기.

각 단계 완료 후 확인하고 다음으로 넘어갈 것.
